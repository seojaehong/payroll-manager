# 바이브코딩 삽질 10선 — AI한테 코딩 시키면서 내가 망한 방법들

> 비개발자 노무사가 Claude Code로 급여관리 시스템을 만들고 있습니다.
> AI가 코드를 잘못 짠 게 아니라, **내가 AI를 잘못 운영해서** 삽질한 이야기입니다.

---

## 1. "그냥 만들어줘"라고 했더니 진짜 그냥 만들어버림

바이브코딩 초기에 가장 많이 한 실수입니다.

> "급여 임포트 기능 만들어줘"

이렇게만 말하면 AI는 진짜로 "일단 돌아가는 무언가"를 만들어 줍니다. 문제는 **어떤 구조로, 어디에, 기존 코드와 어떻게 연결할지**를 내가 결정하지 않으면 AI가 알아서 결정해버린다는 겁니다.

그 결과, 이미 비슷한 기능이 있는 파일을 놔두고 **새 파일을 또 만들고**, 기존 유틸 함수가 있는데 **같은 함수를 또 작성하고**, 나중에 보면 프로젝트에 "거의 같은데 미묘하게 다른" 코드가 두세 벌씩 존재하게 됩니다.

**교훈**: AI한테 시키기 전에 최소한 "어느 파일에 넣을지", "기존에 비슷한 게 있는지" 정도는 물어봐야 합니다. "만들어줘"가 아니라 "만들기 전에 기존 코드 먼저 보여줘"가 먼저입니다.

---

## 2. 컨텍스트가 날아간 걸 모르고 계속 대화함

Claude Code에는 **컨텍스트 윈도우**라는 한계가 있습니다. 대화가 길어지면 앞부분이 잘립니다. 이걸 처음에 몰랐습니다.

한 세션에서 3시간 넘게 작업하면서 "아까 만든 그 함수 수정해줘"라고 했더니, AI가 엉뚱한 파일을 수정합니다. "아까"가 이미 컨텍스트에서 날아간 겁니다.

더 황당한 건, AI가 **"기억이 안 난다"고 말하지 않는다**는 겁니다. 컨텍스트에 없는 내용을 물어보면, 그럴듯하게 추론해서 대답합니다. 그래서 내가 의도한 것과 전혀 다른 코드가 나오는데, 언뜻 보면 맞는 것 같아서 넘어가고... 나중에 버그로 터집니다.

**교훈**: 대화가 길어지면 새 세션을 시작하세요. 그리고 중요한 맥락은 `CLAUDE.md`에 적어두면 매 세션마다 AI가 읽습니다. 기억력을 AI에게 의존하지 마세요. 파일에 적어두세요.

---

## 3. 커밋을 안 하고 10개 파일을 연속으로 수정함

AI가 빠르니까 신나서 계속 시킵니다.

> "이 파일도 고치고, 저 파일도 고치고, 새 기능도 추가하고..."

한 번에 10개 파일을 수정한 다음에 빌드를 돌리면... 에러가 터집니다. 그런데 10개 파일 중 어디서 문제가 생긴 건지 알 수가 없습니다. AI한테 "고쳐줘"라고 하면 또 3개 파일을 건드리면서 새로운 문제가 생기고, 이게 눈덩이처럼 커집니다.

결국 전부 되돌리고 처음부터 다시 했습니다. git stash도 안 해놨으니 수작업으로 되돌리는 것도 삽질이었고요.

**교훈**: AI한테 작업을 시키더라도 **"기능 하나 = 커밋 하나"** 원칙을 지켜야 합니다. 문제가 생기면 `git revert` 한 방이면 되니까요. 바이브코딩의 속도감에 취해서 이걸 잊으면 나중에 10배로 갚습니다.

---

## 4. CLAUDE.md 안 만들어놓고 왜 맨날 같은 말 하냐고 짜증냄

매 세션마다 이렇게 말하고 있었습니다:

> "Firestore에 undefined 넣으면 안 돼, cleanUndefined 써야 해"
> "zustand은 개별 selector로 써야 해, 구조분해 하지 마"
> "주민번호는 replace(/-/g, '') 말고 replace(/[^0-9]/g, '') 써야 해"

3번째 세션부터 내가 짜증이 나기 시작했습니다. "왜 맨날 같은 실수를 하지?"

원인은 간단했습니다. **내가 안 알려줬으니까.** `CLAUDE.md`라는 프로젝트 설명 파일에 이런 규칙을 적어놓으면 매 세션마다 AI가 자동으로 읽는다는 걸 한참 뒤에야 알았습니다.

`CLAUDE.md`를 제대로 세팅한 뒤로는 같은 실수가 거의 사라졌습니다.

**교훈**: AI를 탓하기 전에 내가 규칙을 문서화했는지 점검하세요. `CLAUDE.md`는 AI를 위한 사수 매뉴얼입니다. 신입한테 매뉴얼도 안 주고 "왜 몰라?"라고 하면 그게 누구 잘못인가요?

---

## 5. AI가 만든 코드를 이해 안 하고 다음 기능으로 넘어감

바이브코딩의 가장 달콤한 함정입니다.

AI가 200줄짜리 컴포넌트를 만들어줍니다. 실행해봅니다. 돌아갑니다. "OK 다음!"

한 달 뒤, 그 컴포넌트에 기능을 추가해야 합니다. 코드를 열어봅니다. **전혀 모르겠습니다.** 내가 시킨 건데 내가 모릅니다. AI한테 물어봅니다. 새 세션이라 AI도 모릅니다. 결국 코드를 처음부터 읽어야 합니다.

더 나쁜 건, 이해 없이 쌓인 코드 위에 또 이해 없이 코드를 쌓으면 **아무도 모르는 코드베이스**가 만들어진다는 겁니다. AI도 컨텍스트를 잃으면 기존 구조를 무시하고 새로 만들어버리니까요.

제 프로젝트에서 실제로 벌어진 일입니다:
- `import/page.tsx` (623줄) — 근로자 임포트
- `WagesTab.tsx` (1024줄) — 급여 임포트
- 두 파일에 `parseExcelDate`라는 **같은 이름의 함수가 각각 존재** (로직이 미묘하게 다름)

**교훈**: AI가 코드를 만들면, 최소한 "이 코드 구조 설명해줘"라고 한 번은 물어보세요. 이해하지 않고 넘어간 코드는 반드시 부메랑으로 돌아옵니다.

---

## 6. "dangerously-skip-permissions" — 귀찮으니까 안전장치 꺼버림

Claude Code는 파일 수정이나 명령 실행 전에 허가를 묻습니다. 처음에는 좋았는데, 반복되니까 귀찮아집니다.

> "파일 수정해도 될까요?" → 허용
> "이 명령 실행해도 될까요?" → 허용
> "이 파일도 수정할까요?" → ...귀찮다

그래서 `--dangerously-skip-permissions`를 켰습니다. 이름부터 위험하다고 써있는데요.

물론 대부분의 경우 문제없습니다. 하지만 가끔 AI가 의도치 않게 **엉뚱한 파일을 수정**하거나, **git push를 자동으로 해버리는** 상황이 생길 수 있습니다. 안전장치를 끈 대가는 "그때 왜 꺼놨지..."라는 후회로 돌아옵니다.

**교훈**: 허가 요청이 귀찮으면 `CLAUDE.md`에 허용 범위를 명확히 써놓으세요. "파일 읽기/쓰기는 자동, git push는 물어봐" 이런 식으로요. 안전장치를 끄는 게 아니라 **범위를 조정**하는 겁니다.

---

## 7. 설계 없이 바로 "코드 짜줘"

이게 바이브코딩의 본질적인 함정입니다.

AI가 워낙 빠르게 코드를 써주니까, **설계 단계를 건너뛰게 됩니다.** "일단 만들고 나중에 고치지 뭐."

제 급여 프로그램이 정확히 이 패턴으로 커졌습니다:
1. "급여 계산 만들어줘" → 일단 돌아가는 코드 생성
2. "퇴직금도 추가해줘" → 별도 모듈로 또 생성
3. "임포트 기능도" → 또 별도로 생성
4. "이제 이것들 연결해줘" → 구조가 안 맞아서 대공사

결국 3개월 뒤에 **아키텍처 재설계**를 해야 했습니다. "사업장마다 엑셀 구조가 다른데, 이걸 코드에 하드코딩하지 말고 설정 파일로 빼자" — 이걸 처음부터 생각했으면 3개월의 삽질이 필요 없었습니다.

**교훈**: AI에게 "코드 짜줘" 전에 "이 기능의 구조를 설계해줘"라고 먼저 시키세요. 계획에 5분 쓰면 구현에서 5시간을 아낍니다. 바이브코딩은 속도가 빠른 만큼, **잘못된 방향으로도 빠르게 갑니다.**

---

## 8. Phase 뛰어넘기 — 2단계를 건너뛰고 3단계 먼저 함

체계적으로 하겠다고 Phase를 나눴습니다:
- Phase 1: 설정 구조 만들기 (TypeScript)
- Phase 2: 기존 코드 통합 (import 엔진)
- Phase 3: Python 처리 엔진

그런데 Phase 1을 끝내고 나니, Phase 2(코드 통합)가 지루해 보였습니다. "Phase 3가 더 재밌겠다!"

Phase 2를 건너뛰고 Phase 3을 먼저 했습니다. Python 엔진이 멋있게 완성됐습니다. 하지만 Phase 2를 안 했으니까:
- TypeScript 쪽에는 여전히 중복 코드가 남아있고
- Python과 TypeScript가 공유해야 할 config 구조가 약간 어긋나 있고
- 나중에 Phase 2를 할 때 Phase 3 코드도 같이 수정해야 하는 상황

**교훈**: 순서를 정했으면 지키세요. AI가 빠르다고 "일단 재밌는 것 먼저"는 기술 부채를 재미있게 만드는 방법입니다.

## 9. 한 세션에서 너무 많은 걸 하려 함

이것도 반복되는 실수입니다.

한 번 앉으면 "오늘 다 끝내야지"라는 마음으로 3-4시간을 연달아 작업합니다. AI 대화가 수십 턴을 넘어가면:

- 컨텍스트 윈도우가 포화되어 앞부분이 잘림
- AI가 초반에 합의한 구조를 잊어버림
- 같은 파일을 여러 번 수정하면서 충돌 발생
- 내 판단력도 떨어져서 AI 출력을 검증 없이 수락

제가 이걸 실감한 게, 한 세션에서 작업하다가 컨텍스트가 꽉 차서 **세션이 자동으로 요약 압축된 적이 있습니다.** AI가 "이전 대화 요약입니다..."라고 하면서 세부사항이 날아갔는데, 그 세부사항 중에 아직 안 끝난 작업이 있었거든요.

**교훈**: 한 세션에 한 가지 주제. "오늘 Phase 1만 한다" 이런 식으로 범위를 잡으세요. 세션이 길어지면 커밋하고, 새 세션에서 이어서 하세요. AI의 기억력은 무한하지 않습니다.

## 10. AI를 "개발자"로 착각함

가장 근본적인 삽질입니다.

AI는 개발자가 아닙니다. **아무리 코드를 잘 짜도 AI는 도구**입니다. 도구는 시키는 대로 합니다. 시키지 않은 건 안 합니다.

- AI는 "이 기능이 기존 기능과 충돌할 수 있어요"라고 **먼저 말해주지 않습니다**
- AI는 "이 구조는 나중에 문제가 될 것 같아요"라고 **경고하지 않습니다** (시키면 합니다)
- AI는 "어제 이 부분을 이미 만들었는데요"라고 **기억하지 못합니다** (새 세션이면)

프로젝트의 **방향, 구조, 우선순위를 결정하는 건 나**입니다. AI는 내가 정한 방향으로 빠르게 달려주는 역할이지, 방향을 정해주는 역할이 아닙니다.

제가 이걸 깨닫고 나서 바뀐 점:
- 기능 요청 전에 "기존에 비슷한 거 있는지 먼저 검색해줘"
- 코드 작성 전에 "설계 먼저 보여줘, 기존 구조와 충돌 없는지 확인해줘"
- 작업 후 "방금 만든 것과 기존 코드 사이에 중복이 없는지 체크해줘"

이 세 마디를 추가한 것만으로 삽질이 절반으로 줄었습니다.

**교훈**: 바이브코딩에서 AI는 최고의 시공 업체입니다. 하지만 설계도 없이 "알아서 지어주세요"라고 하면, 아무리 실력 좋은 업체라도 결과물이 엉망이 됩니다. **건축주(나)의 책임입니다.**

---

## 마무리: 바이브코딩은 운전과 비슷하다

자동차가 아무리 좋아도 운전을 못하면 사고가 납니다.

바이브코딩도 마찬가지입니다. AI라는 엔진이 아무리 강력해도, 방향을 잡고 브레이크를 밟는 건 사람의 몫입니다.

제가 3개월간 배운 바이브코딩 생존 규칙:

1. **시키기 전에 물어봐라** — "기존에 비슷한 거 있어?"
2. **설계 먼저, 코드 나중** — "구조 설계해줘, 그다음 코드 짜줘"
3. **자주 커밋해라** — 기능 하나 = 커밋 하나
4. **CLAUDE.md를 관리해라** — AI의 장기 기억은 파일에 있다
5. **한 세션에 한 주제** — 욕심부리면 컨텍스트가 날아간다
6. **AI 출력을 이해해라** — 이해 안 되면 설명을 시켜라
7. **순서를 지켜라** — 재밌는 것 먼저 하면 지루한 빚이 쌓인다

노무사가 AI로 급여 프로그램을 만들 수 있는 시대입니다. 대단한 시대인 건 맞는데, **삽질도 AI 속도로 한다**는 걸 잊지 마세요.
